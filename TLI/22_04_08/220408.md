### 디자인 패턴이 뭔가요?

* 디자인 패턴이란 기존 환경 내에서 반복적으로 일어나는 문제들을 어떻게 풀어나갈 것인가에 대한 일종의 솔루션이다. 
  *  1. 문제발생 -> 2. 기존 디자인 패턴으로 해결 가능한지 판단
    * 2-1. 해결 가능할 경우 -> 기존 패턴으로 빠르고 안전하게 해결
    * 2-2. 해결 불가능한 경우 -> 문제를 파악하고 해결한뒤, 새로운 디자인 패턴으로 저장. 

* 디자인 패턴의 3가지 종류. 

1. 생성(Creatinoal) 패턴
2. 구조(Structural) 패턴
3. 행동(Behavioral) 패턴

### 생성 패턴   (class 구현 느낌인듯)

* 객체 생성에 관련된 패턴
* 객체의 생성과 조합을 캡슐화하여 특정 객체가 생성, 변경되어도 프로그램 구조에 영향을 크게 받지 않도록 유연성을 제공함. 

### 구조 패턴 (add 함수 느낌인듯)

* 클래스나 객체를 조합해 더 큰 구조를 만드는 패턴
* ex) 서로 다른 인터페이스를 지닌 2개의 객체를 묶어 단일 인터페이스를 제공하거나 겍체들을 묶어 새로운 기능을 제공하는 패턴.

### 행위 패턴 , (division 느낌인듯) 

* 객체나 클래스 사이의 알고리즘이나 책임분배에 관련된 패턴
* 한 객체가 혼자 수행할 수 없는 작업을 여러개의 객체로 어떻게 분배하는지, 또 그렇게 하면서 객체 사이의 결합도 최소화 하는 것에 중점을 둔다.

### what is the Data Coupling(자료결합도)?

* 먼저 결합도(Coupling)과 응집도(Cohension)란?

#### 결합도(Coupling)

* 결합도는 서로 다른 모듈 간에 상호 의존하는 정도 또는 연관된 관계를 의미한다. 
* 결합도가 높은 클래스는 다른 클래스와 연관된 정도가 높다.
* 따라서 해당 클래스를 변경하면 연관된 클래스도 변경해야 하며, 다른 코드에서 클래스를 재사용하기도 어렵다. 

---
* 여기서 한번 더 정리하는 모듈의 개념

> 모듈(Module) & 모듈화(Modularization)
> > 먼저 모듈화란 소프트웨어를 각 기능별로 나누는 것을 말한다.
> > 이때, 각각 기능별로 나누어진 스포트웨어를 *모듈(Module)* 이라고 하며, 
> > 서브루틴(Subroutine), 소프트웨어 내의 프로그램 또는 작업 단위 등의 의미로 사용된다. 
>
> 좋은 모듈화의 기준, ( 대충 막 나눈다고 "모듈화했어요!" 가 아님 )
> > 좋은 모듈화는 *목적*에 맞는 *기능*을 초점으로 모듈을 구분하는 것이다. 
> > 즉 *독립성*이 높은 모듈일 수록 좋다!

* 이때 모듈의 *독립성*은 모듈의 결합도(Coupling)와 응집도(Cohension)로 측정한다. 
  * 결론적으로 모듈의 독립성을 높이기 위해서는 모듈 간의 상호 의존 정도를 나타내는 결합도를 낮추고 모듈이 독립적으로 자체 기능만을 수행하도록 응집도를 높여야 한다.

---

#### 중간정리, 

1. 결합도와 응집도는 좋은 모듈화를 구분하는 척도가 된다. 
2. 결합도가 높을 수록 모듈간 상호 의존적인 관계가 높음을 의미한다. 
3. 반면 응집도가 높을 수록 모듈 내부 요소들의 체계가 응집되어있으며, 타 모듈과 분리되어 기능을 수행함을 나타낸다.

#### => 모듈화의 수준은 결합도에 반비례하며, 응집도에 비례한다.

### 그래서 자료 결합도(Data Coupling)가 뭐에요?

* 가장 낮은 결합도를 갖는다. 가장 좋은 형태이며, 모듈끼리 단순히 파라미터 등을 통해 데이터를 주고받는 경우이다. 
  * 파라미터를 통해 데이터를 주고받는게 왜 결합도가 낮은(모듈이 잘 분리되어있음)을 의미하는 것일까? (예측해보자.)
  * => 모듈(편의상 함수라고 가정하겠다.), 함수 간 데이터 공유를 외부 메모리(변수)를 사용해 진행하기 때문에, 
  * 직접적인 함수 내부 주소값의 메모리 충돌이 일어나지 않는다. 
* 여기서 주고받는 데이터는 모듈의 기능 수행에 있어서 로직을 제어하거나 하지 않는 순수한 자료형 요소이다. 
* 또한 하나의 모듈을 변경하더라도 다른 모듈에는 영향을 끼치지 않는 결합 형태이다. (좋네요~)

### 스탬프 결합도(Stamp Coupling)

* 두 모듈이 동일한 자료 구조를 참조하는 형태의 결합도이다.
* 즉, 모듈간의 인터페이스로 배열 또는 오브젝트 등이 전달되는 경우를 말한다.
* 자료 구조의 형태가 변경되면 그것을 참조하는 모든 모듈에 영향을 주며 변경되는 필드를 실제로 참조하지 않는 모듈에도 영향을 준다. 
  * 흠... 함수 내부 재귀호출과 지역변수의 재활용 등이 예시가 될 것같다. c++에서는 독립된 모듈을 만들어 길지 않은 코드로 특정 component를 제어하는 경우에는 크게 문제가 될 것같지 않지만,,, 
  * 만일 코드가 길어지거나, 타입 명시에 엄격하지 않은 언어를 사용해 코드를 작성할 경우, 오버플로가 발생하거나, 예상치않은 오류가 발생할 가능성이 있을 것같아 지양하는 것이 좋을 것같다. ㅇㅇ

### 제어 결합도(Control Coupling)

* 어떤 모듈이 다른 모듈 내부의 논리적인 흐름을 제어하는 요소를 전달하는 경우를 말한다. 
  *  아 이건 좀 많이 위험한 코드 작성일 것같다.
  *  함수 내에서 조건문을 추가해 각기 다른 함수 혹은 변수를 호출해 사용하는 경우가 해당된다.
  *  이 경우, 1. 조건문에서 에러발생, 2. 조건문 이후 함수 혹은 변수 호출시 문제발생, 3. 적절한 호출 외 garbage 값의 제대로된 초기화가 이뤄지지 않은 경우,, 등 여러 문제가 있을 수 있기 때문에,,, 지양하자 ㅇㅇ

### 외부 결합도(External Coupling)

* 모듈이 외부에 있는 다른 모듈 또는 데이터를 참조할 때의 결합 형태를 말한다. 
* 외부 결합도는 모듈이 외부의 데이터, 통신 프로토콜 등을 공유할 때 발생한다. 데이터를 참조 또는 공유하는 결합 형태가 공통 결합과 비슷하지만 참조하는 데이터가 외부에 위치한다는 점에서 차이가 있다. 

### 공통 결합도(Common Coupling)

* 여러 모듈이 하나의 데이터 영역을 참조하여 사용하느 형태다. 전역 변수(global variable)를 예로 들 수 있다. 
  * 아 이건 인정, 전역 변수를 사용할 경우 코드를 조금 더 간결하게 만들 수 있다는 장점이 있지만, 
  * 만일 협업을 하거나, 무분별한 전역 데이터의 참고가 이뤄질 경우 많은 문제를 일으킬 수 있다. ㅇㅇ 

### 내용 결합도(content Coupling)

* 가장 높은 결합도를 갖는다. 가장 좋지 않은 결합 형태이다.
* 어떤 모듈이 사용하려는 다른 모듈의 내부 기능과 데이터를 직접 참조하는 경우이다.
* 다른 모듈의 로컬 데이터에 접근하는 경우처럼 사용하고자 하는 모듈의 내용(코드)를 알고 있어야한다.
* 모듈의 변경이 발생하는 경우 이를 참조하는 모듈의 변경이 반드시 필요하게 되므로 가장 좋지 않은 결합이다. 
  *  결론: 쓰지말자 먹지말자 내용 결합도 ㅇㅇ

## 응집도(Cohension)가 뭐에요 선생님? 

* 응집도는 한 모듈 내부의 처리 요소들이 서로 관련되어 있는 정도를 말한다.
* 즉, 모듈이 독립적인 기능을 수행하는지 또는 하나의 기능을 중심으로 책임이 잘 뭉쳐있는지를 나타내며 모듈이 높은 응집도를 가질 수록 좋다.
  * => 응집도는 결합도에 반대되는 개념이라고 가볍게 생각해도 좋다. (정확히는 아님)

### 기능적 응집도(Functional Cohension)

* 응집도가 가장 높은 경우이며, 가장 좋은 형태이다.ㄷ
* 모듈 내의 모든 요소들이 하나의 기능을 수행하기 위해 구성된 경우를 나타낸다. 
  * ex) 코사인(cosine)과 같은 삼각함수를 계산하는 기능을 모아둔 함수 등이라고 한다. ㅇㅇ 

### 순차적 응집도(Sequential Cohension)

* 한 요소의 출력이 다른 요소의 입력으로 사용되는 형태이다. 어떤 모듈이 특정 파일을 읽고 처리하는 기능을 하는 등과 같다고 한다. 
  * 재귀함수 생각하면 될듯 ㅇㅇ

### 통신적 응집도(Communicational Cohension)

* 모든 요소들이 동일한 입력 또는 출력 데이터를 사용하여 서로 다른 기능을 수행하는 경우다.
  * 어 그럼 순차적 응집도랑 뭐가다른가요?
  * => 통신적 응집도는 처리 순서가 중요하지 않다고 한다. ㅇㅇ 흠.. 그럼 미리 만들어진 지역함수를 파라미터로 재귀호출하는 함수를 생각하면 이해가 편할 것같다. 

### 절차적 응집도(Procedural Cohension)

* 모듈내에서 여러 개의 기능 요소가 순차적으로 수행되지만 다음 기능 요소로 데이터가 아닌 흐름 제어 요소가 전달되는 경우, 
  * 예를 들면, 파일을 읽을 때 접근 허가를 확인한 후에 파일을 읽는 형태 등이다. 

### 일시적 응집도(Temporal Cohension)

* 각 기능 요소들이 순서에 상관없이 특정 시점에 반드시 수행되는 경우다. 
  * 예를 들면, 프로그램이 구동될 때 초기화 시키는 모듈이나 예외 상황이 발생했을 때 오류 로그를 개발자에게 전송하는 기능 등 순서에 상관없는 경우를 말한다. 

### 논리적 응집도(Logical Cohension)

* 유사한 성격을 갖거나 특정 형태로 분류되는 처리 요소들을 모듈을 구성하며 논리적으로 비슷한 기능을 수행하지만 서로의 관계는 밀접하지 않은 형태
  * switch case문으로 조건별 수행 함수를 분류해둔 함수 ㅇㅇ 

### 우연적 응집도(Coincidental Cohension)

* worst case
* 모듈 내부의 각 구성 요소들이 아무런 관련 없이 구성된 형태
  * 어 그러면 논리적 응집도와 뭐가다른가요?
  * => 각 구성 요소들 간의 유사한 성격이나 형태가 없이, 전혀 무관한 함수들 간의 집합,,
  * 이 경우, 모듈 수정이 사이드 이팩트를 발생시킬 가능성이 매우 높다... :(
